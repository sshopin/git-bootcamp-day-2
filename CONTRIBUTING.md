# Contributing to git-bootcamp-day-2

Спасибо, что хотите внести вклад. Несколько правил, чтобы процесс шёл гладко.

## Важно!

Внося свой вклад в этот проект, вы:

* Соглашаетесь, что вы являетесь автором 100% контента.
* Соглашаетесь, что у вас есть необходимые права на контент
* Соглашаетесь, что вы получили необходимые разрешения от вашего работодателя для внесения вклада (если применимо).
* Соглашаетесь, что предоставляемый вами контент может предоставляться по лицензии(ям) проекта.
* Соглашаетесь, что если вы не являетесь автором 100 % контента, соответствующие лицензии и авторские права будут добавлены вместе с любым другим необходимым указанием авторства.

## Как сообщить об ошибке

1. Поищите в открытых issues — возможно, такая уже есть.
2. Если нет — создайте новый issue. Опишите:
   - что ожидали увидеть;
   - что получили на самом деле;
   - шаги воспроизведения;
   - версию Python и ОС.

## Как предложить улучшение

1. Сначала откройте issue с описанием идеи — лучше обсудить до написания кода.
2. После одобрения — форкните репозиторий, создайте ветку, сделайте PR.

Пожалуйста, создавайте issues только для отчетов об ошибках или запросов функций. Вопросы, обсуждающие любые другие темы, могут быть закрыты мейнтейнерами проекта без дополнительных объяснений.

Не создавайте issues с изменением зависимостей, пока не будет обнаружена ошибка и вы не сможете продемонстрировать, что она влияет на этот проект.

## Процесс работы с кодом

```bash
# Форкаем и клонируем
git clone git@github.com:YOUR-USERNAME/git-bootcamp-day-2.git
cd sample-cli

# Создаём ветку под изменение
git checkout -b feat/short-description

# Делаем изменения, коммитим
git commit -m "feat: add short description"

# Пушим и открываем PR
git push origin feat/short-description
```

## Стиль кода

- Python: `ruff check .` должен проходить без ошибок.
- Тесты обязательны для новых фич.
- Сообщения коммитов — на ваше усмотрение, но желательно понятные.

## Code review

- PR проходит ревью минимум одного мейнтейнера.
- Ответ обычно в течение 2–3 рабочих дней.
- Мы стараемся давать конструктивный фидбек, не «переделай всё».

## Closing issues
Авторы или мейнтейнеры проекта git-bootcamp-day-2 могут закрыть issue в любое время. Обычно, но не исключительно, issues закрываются, когда:
- Проблема решена.
- Разработчики проекта определили, что issue не соответствует целям и задачам проекта.
- issue явно является дубликатом, и в этом случае дублирующая issue будет связана.
- Обуждение явно исчерпало себя.
